# Trivy DB caching approach:
# The Trivy action downloads the vulnerability DB to ~/.cache/trivy/db on first run.
# Subsequent runs check for a local DB and only download if outdated.
# By caching this directory with actions/cache, we avoid repeated downloads.
#
# Cache key strategy: trivy-db-${{ runner.os }}-${{ hashFiles('**/trivy-cron.yml') }}
# This ensures the cache is refreshed when the workflow changes.
#
# The cron job runs `trivy image --download-db-only` to fetch the latest DB,
# then caches it for subsequent workflow runs.

# security-events: write (optional — could upload SARIF of DB version)

Create `.github/workflows/trivy-cron.yml` with the following content:

```yaml
name: Trivy DB Pre-Warm

on:
  schedule:
    # Daily at 06:00 UTC — before most CI builds start
    - cron: '0 6 * * *'
  workflow_dispatch: # Allow manual trigger for testing

permissions:
  contents: read

jobs:
  prewarm-db:
    name: Pre-warm Trivy DB
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v6

      - name: Cache Trivy DB
        uses: actions/cache@v5
        with:
          path: ~/.cache/trivy
          key: trivy-db-${{ runner.os }}-${{ hashFiles('.github/workflows/trivy-cron.yml') }}
          restore-keys: |
            trivy-db-${{ runner.os }}-

      - name: Run Trivy to download latest DB
        uses: aquasecurity/trivy-action@v0.35.0
        with:
          # Download only the vulnerability database (no image scan)
          # This populates ~/.cache/trivy/db with the latest DB
          image-ref: alpine:latest
          severity: CRITICAL,HIGH
          exit-code: "0"
          ignore-unfixed: true
          vuln-type: os,library
          timeout: 10m

      - name: Verify DB cache
        run: |
          echo "Trivy DB cache status:"
          ls -la ~/.cache/trivy/db/ 2>/dev/null || echo "DB cache directory not found"
          echo "Cache key: trivy-db-${{ runner.os }}"

      - name: Report success
        if: success()
        run: echo "Trivy DB pre-warmed successfully at $(date -u)"

      - name: Report failure
        if: failure()
        run: |
          echo "WARNING: Trivy DB pre-warm failed at $(date -u)"
          echo "Next build may hit rate limits on DB download"
          exit 0 # Don't fail the workflow — DB may still be cached from previous run
```

Key design decisions implemented:
- D-08: Separate workflow file (clean separation of concerns)
- D-09: Cron at `0 6 * * *` (06:00 UTC daily), plus `workflow_dispatch` for manual testing
- Cache key includes OS and workflow hash for proper invalidation
- `exit-code: "0"` for the DB download step — we don't want to fail if alpine scan finds vulns
- `ignore-unfixed: true` reduces noise from base images
- Failure step reports warning but doesn't fail the workflow (DB may still be cached)
- `timeout: 10m` prevents hanging on slow DB downloads
- Permissions: `contents: read` only (minimal — no security-events needed for cron)
- REQ-SEC-001: Cron job ensures Trivy DB is fresh, avoiding rate limits on first build of day
